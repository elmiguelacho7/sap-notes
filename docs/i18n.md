# Internationalization (i18n) — Ribbit

Internal reference for how Ribbit handles translations with **next-intl** without locale-based routing.

---

## 1. Overview

- Ribbit uses **[next-intl](https://next-intl.dev)** for message catalogs and React/server APIs (`useTranslations`, `getTranslations`, `getMessages`, etc.).
- **There is no locale in the URL.** Paths stay `/`, `/dashboard`, `/projects`, etc. — never `/en/...` or `/es/...`.
- **Locale is resolved from a cookie:** `NEXT_LOCALE` (see `LOCALE_COOKIE_NAME` in `i18n/config.ts`).
- **Default locale is `en`.** If the cookie is missing or invalid, the app falls back to English.

---

## 2. Key Principles

| Principle | Detail |
|-----------|--------|
| Clean URLs | No locale prefix on any route. |
| Preference-driven locale | User language is stored in `NEXT_LOCALE`; server reads it per request. |
| Parity | Every user-visible key must exist in **both** `messages/en/` and `messages/es/`. |
| No stray literals | Prefer messages for UI copy; avoid hardcoded strings in components you touch. |
| Messages only | next-intl is **not** used for routing — no `createMiddleware` from `next-intl/middleware`, no `app/[locale]/`. |

---

## 3. Architecture

### Flow

1. User picks a language in the shell (**User menu**). The client sets the `NEXT_LOCALE` cookie and calls `router.refresh()`.
2. On the next request, **`i18n/request.ts`** (next-intl `getRequestConfig`) reads the cookie via `cookies()` from `next/headers`.
3. The value is validated (`isAppLocale` in `i18n/config.ts`); invalid or missing → **`en`**.
4. **`loadMessages`** imports the four JSON bundles for that locale and merges them into one object with top-level namespaces: `common`, `auth`, `sidebar`, `dashboard`.
5. **`app/layout.tsx`** calls `getLocale()` and `getMessages()`, then wraps the tree in **`NextIntlClientProvider`** so client components can use `useTranslations`.

**Middleware:** `middleware.ts` only refreshes the Supabase session. It does **not** participate in locale detection.

## 4. File structure

```
messages/
  en/
    common.json
    auth.json
    sidebar.json
    dashboard.json
  es/
    common.json
    auth.json
    sidebar.json
    dashboard.json

i18n/
  config.ts      # LOCALES, DEFAULT_LOCALE, LOCALE_COOKIE_NAME, isAppLocale
  request.ts      # getRequestConfig: cookie → locale → merged messages

app/
  layout.tsx      # NextIntlClientProvider, getLocale, getMessages
```

**Adding a new message file** (e.g. `notes.json`): place it under `messages/en/` and `messages/es/`, then update **`loadMessages`** in `i18n/request.ts` to import it and add a namespace key (e.g. `notes: notes.default`).

---

## 5. Adding new translations

1. Add the key in **`messages/en/<namespace>.json`** (English is the source of truth for copy).
2. Add the **same key path** in **`messages/es/<namespace>.json`** with the Spanish string.
3. In a **client** component: `useTranslations("<namespace>")`.
4. In a **server** component: `getTranslations({ namespace: "<namespace>" })` (or nested namespace as supported by next-intl).
5. Replace the hardcoded string with `t("key")` or `t("section.key")`.

**Example (client):**

```tsx
"use client";

import { useTranslations } from "next-intl";

export function Greeting() {
  const t = useTranslations("dashboard");
  return <span>{t("greeting.morning")}</span>;
}
```

**Example (server):**

```tsx
import { getTranslations } from "next-intl/server";

export default async function Page() {
  const t = await getTranslations("dashboard");
  return <h1>{t("greeting.morning")}</h1>;
}
```

## 6. Naming conventions

- Use **semantic, stable keys**: `button.create`, `search.placeholder`, `errors.loadFailed`.
- Prefer **no more than 2–3 levels** of nesting (e.g. `kpi.openTickets`, `activity.eventTypes.project_created`).
- Avoid opaque names: `text1`, `labelA`, `msg` — they do not scale or review well.
- Keep namespaces aligned with **domains** (`auth`, `sidebar`, `dashboard`, `common`) unless you introduce a new JSON file + loader entry.

---

## 7. Language switcher

- Implemented in the **user menu** (`components/UserMenu.tsx`).
- Writes **`NEXT_LOCALE`** (values: `en` | `es`) with a path-wide cookie (see `setLocaleCookie` in that file).
- Calls **`router.refresh()`** so the next RSC pass picks up the new cookie **without changing the URL**.

---

## 8. What not to do

- **Do not** add locale-prefixed routes (`/en/dashboard`, `/es/login`, etc.).
- **Do not** reintroduce **next-intl routing middleware** (`createMiddleware` + `defineRouting`) unless product explicitly moves to segment-based i18n.
- **Do not** hardcode user-visible strings in new or heavily edited UI.
- **Do not** ship keys that exist only in EN or only in ES — keep catalogs in sync.

---

## 9. Future improvements (optional)

- **Type-safe messages:** generate or maintain an `IntlMessages` type (e.g. from JSON) to catch missing keys at compile time.
- **More locales:** extend `LOCALES` / `isAppLocale`, add `messages/<locale>/`, and extend `loadMessages` imports.
- **Extraction / lint:** tooling to find missing keys or unused keys across `en` / `es`.
